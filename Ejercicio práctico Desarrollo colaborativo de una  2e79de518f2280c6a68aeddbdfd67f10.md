# Ejercicio práctico: Desarrollo colaborativo de una landing page con control profesional en GitHub

### Propósito del ejercicio

Simular un entorno real de trabajo en equipo donde varios desarrolladores colaboran sobre un mismo producto sin interferencias, usando control de versiones, coordinación explícita y criterios técnicos compartidos. El foco no es solo el resultado visual, sino **el proceso profesional de colaboración**.

## 1. Enunciado del ejercicio

El grupo completo (7 alumnos) debe desarrollar **una landing page completa, funcional y coherente**, trabajando de forma simultánea sobre **un único repositorio público compartido**.

Cada alumno será responsable de **una parte concreta del proyecto**, trabajando de forma independiente, pero siguiendo normas comunes para que el resultado final sea un producto integrado, estable y mantenible.

El trabajo debe realizarse exclusivamente mediante Git y GitHub, sin intercambios de archivos externos ni trabajo “offline” que luego se suba sin control.

## 2. Objetivos de aprendizaje

- Entender GitHub como **herramienta de coordinación**, no solo de almacenamiento.
- Aprender a trabajar con **ramas**, **commits claros** y **Pull Requests**.
- Evitar conflictos de código mediante **reparto explícito de responsabilidades**.
- Aprender a leer y respetar el trabajo de otros desarrolladores.
- Integrar partes independientes en un único producto funcional.
- Simular un flujo de trabajo profesional básico de equipo.

## 3. Organización del equipo (roles y responsabilidades)

La asignación final de personas a roles debe hacerla el propio grupo, pero **los roles están definidos y no son opcionales**.

### Roles propuestos

1. **Responsable de estructura base**
    - HTML principal.
    - Estructura semántica global.
    - Integración final de secciones.
2. **Responsable de header y navegación**
    - Header visual.
    - Menú de navegación funcional.
    - Enlaces internos correctos.
3. **Responsable de sección Hero**
    - Bloque principal de impacto.
    - Título, subtítulo, llamada a la acción.
4. **Responsable de sección de contenido**
    - Features, servicios o beneficios.
    - Estructura clara y legible.
5. **Responsable de sección visual/testimonial**
    - Cards, testimonios, bloques visuales.
    - Coherencia con el diseño general.
6. **Responsable de footer**
    - Footer completo.
    - Información secundaria, enlaces, copyright.
7. **Responsable de integración y control**
    - Revisión de Pull Requests.
    - Resolución de conflictos.
    - Verificación de que todo funciona al final.

Nadie trabaja directamente sobre la rama principal sin revisión.

## 4. Requisitos funcionales del proyecto

La landing page debe cumplir **todos** los siguientes puntos:

- Una única página HTML.
- Navegación interna funcional (anclas).
- Estructura semántica correcta (`header`, `main`, `section`, `footer`).
- Diseño coherente entre secciones.
- Contenido realista (no lorem ipsum genérico).
- Código legible y bien indentado.
- Proyecto ejecutable abriendo el HTML en local.

No se evalúa creatividad extrema. Se evalúa **orden, coherencia y colaboración**.

## 5. Requisitos técnicos mínimos

- Repositorio público en GitHub.
- Rama `main` protegida (conceptualmente).
- Una rama de trabajo por alumno.
- Commits pequeños, descriptivos y frecuentes.
- Uso obligatorio de Pull Requests para integrar cambios.
- Resolución consciente de conflictos si aparecen.

No se permite:

- Subir todo el trabajo en un solo commit.
- Trabajar directamente en `main`.
- Copiar y pegar el trabajo de otro compañero sin coordinación.

## 6. Análisis y especificación básica del diseño

### Estructura base esperada

- Header fijo o estático con navegación.
- Hero section con mensaje principal.
- Sección de contenido principal (servicios / beneficios).
- Sección secundaria (testimonios, ventajas, bloques visuales).
- Footer completo.

### Criterios de coherencia

- Tipografía consistente.
- Espaciados similares entre secciones.
- Estilo visual uniforme.
- Lenguaje textual coherente (no parecer siete páginas distintas).

## 7. Criterios de evaluación

Se evaluará:

- Uso correcto de Git y GitHub.
- Respeto de roles y responsabilidades.
- Calidad de los commits.
- Gestión de ramas.
- Integración final sin roturas.
- Capacidad de coordinarse sin intervención constante.

El resultado final importa, pero **el historial del repositorio importa más**.
